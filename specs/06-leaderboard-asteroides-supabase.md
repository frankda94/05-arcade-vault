# 06 — Leaderboard real de Asteroides con tablas `games` y `scores` en Supabase

**Estado:** Implementado
**Dependencias:** [05-asteroides](05-asteroides.md) (juego jugable y botón "GUARDAR PUNTUACIÓN" en `GamePlayer`), [01-pantallas-visuales](01-pantallas-visuales.md) (leaderboards en `/juego/[id]` y `/salon-fama`)
**Fecha:** 2026-06-11

**Objetivo:** Crear en Supabase una tabla de juegos (`games`, con una fila inicial para `"asteroides"`) y una tabla de puntuaciones (`scores`) para persistir partidas reales, conectando el botón "GUARDAR PUNTUACIÓN" del Reproductor de Asteroides a una inserción real y mostrando ese leaderboard real en `/juego/asteroides` y en la pestaña "ASTEROIDES" de `/salon-fama`.

## Alcance

**Incluye:**

- Migración SQL en Supabase (aplicada vía `apply_migration`) que crea:
  - Tabla `games`: `id` (text, PK), `title` (text), `created_at` (timestamptz, default `now()`).
  - Tabla `scores`: `id` (bigint identity, PK), `game_id` (text, FK → `games.id`), `player_name` (text), `score` (integer), `created_at` (timestamptz, default `now()`).
  - RLS habilitado en ambas tablas con políticas públicas de `select` (todos) e `insert` (todos) — sin `update`/`delete` públicos.
- Seed dentro de la misma migración:
  - Una fila en `games`: `{ id: "asteroides", title: "ASTEROIDES" }`.
  - ~10-12 filas iniciales en `scores` para `game_id = "asteroides"`, con nombres tomados de `PLAYERS` (`lib/data.ts`), puntuaciones y fechas de ejemplo, para que el leaderboard no se vea vacío.
- `/juego/asteroides` (`app/juego/[id]/page.tsx`, server component): para `id === "asteroides"`, consulta `scores` (Supabase server client) filtrando `game_id = "asteroides"`, ordenando por `score` descendente, límite 10, en vez de `seededScores`. Para el resto de juegos, sin cambios (sigue `seededScores`).
- `/salon-fama` (`app/salon-fama/page.tsx`): cuando la pestaña activa es `"asteroides"`, el podio y la tabla muestran los datos reales de `scores` (consulta vía Supabase browser client en un `useEffect`), en lugar de `seededScores`. Para el resto de pestañas, sin cambios.
- `GamePlayer.tsx`: al pulsar "GUARDAR PUNTUACIÓN" cuando `game.id === "asteroides"`, se inserta una fila real en `scores` (`game_id: "asteroides"`, `player_name: name`, `score: displayScore`) vía Supabase browser client. Se muestra el toast "PUNTUACIÓN GUARDADA" tanto si la inserción tiene éxito como si falla (error solo logueado en consola con `console.error`), siguiendo el mismo patrón que el formulario de contacto (spec 04). Para el resto de juegos, "GUARDAR PUNTUACIÓN" sigue siendo solo visual (sin escritura en BD).

**No incluye:**

- Autenticación real ni atribución de jugador vía sesión — se mantiene el input manual de iniciales ya existente (máx. 10 caracteres, default "INVITADO").
- Tablas/leaderboards reales para otros juegos del catálogo (`bloque-buster`, `caída`, etc.) — siguen 100% mock con `seededScores`. Si en el futuro se portan más motores reales, sus filas en `games`/`scores` quedarían para specs futuros.
- Validación anti-cheat, rate limiting o límites de puntuación — cualquiera con la clave pública puede insertar cualquier valor.
- Edición o borrado de puntuaciones (sin políticas RLS de `update`/`delete`), ni paginación del leaderboard.
- Actualizar `game.best`/`game.plays` de la entrada `"asteroides"` en `lib/data.ts` con datos derivados de `scores` — esos campos siguen siendo el valor estático mock.
- Refrescar automáticamente el leaderboard de `/juego/asteroides` o `/salon-fama` en tiempo real tras guardar (el jugador vería su score nuevo al navegar/recargar esas páginas, no instantáneamente desde el Reproductor).
- Tests automatizados (no hay test runner configurado).

## Modelo de datos

**Tablas nuevas en Supabase** (migración SQL):

```sql
create table public.games (
  id text primary key,
  title text not null,
  created_at timestamptz not null default now()
);

create table public.scores (
  id bigint generated by default as identity primary key,
  game_id text not null references public.games(id),
  player_name text not null,
  score integer not null,
  created_at timestamptz not null default now()
);

alter table public.games enable row level security;
alter table public.scores enable row level security;

create policy "games_select_public" on public.games
  for select to anon, authenticated using (true);

create policy "scores_select_public" on public.scores
  for select to anon, authenticated using (true);

create policy "scores_insert_public" on public.scores
  for insert to anon, authenticated with check (true);
```

```typescript
export interface ScoreRow {
  rank: number;
  name: string;
  score: number;

```

Esta forma `ScoreRow` ya existe en `lib/data.ts` (la usa `seededScores`); las consultas reales a `scores` se mapean al mismo shape para que `/juego/asteroides` y `/salon-fama` puedan renderizar ambas fuentes (mock y real) con el mismo componente/markup, asignando `rank` según el orden de la consulta (`order by score desc`).

No se modifica el tipo `Game` ni la entrada `"asteroides"` de `GAMES` en `lib/data.ts`.

## Plan de implementación

1. **Crear y aplicar la migración SQL** (`apply_migration` vía MCP de Supabase) con las tablas `games`/`scores`, las políticas RLS y los datos de seed definidos en la sección anterior. Sin cambios de código todavía; el sistema sigue funcionando igual (Biblioteca, `/juego/asteroides`, `/salon-fama` siguen usando `seededScores`).

2. **Crear `lib/leaderboard.ts`** con:
   - `mapScoreRows(rows): ScoreRow[]` — convierte filas `{ player_name, score, created_at }` de `scores` al shape `ScoreRow` (`rank` asignado por posición, `date` formateada `DD/MM/YYYY` desde `created_at`).
   - `getAsteroidesScores(supabase): Promise<ScoreRow[]>` — `select` en `scores` con `game_id = "asteroides"`, `order by score desc`, `limit 10`, mapeado con `mapScoreRows`. Recibe el cliente de Supabase (server o browser) ya creado.
   - `saveAsteroidesScore(supabase, playerName, score): Promise<void>` — `insert` en `scores` (`game_id: "asteroides"`, `player_name`, `score`); no lanza si falla, solo `console.error`.
     El sistema sigue funcional — el helper aún no se usa en ninguna página.

3. **Modificar `app/juego/[id]/page.tsx`**: si `id === "asteroides"`, usar `createClient` de `utils/supabase/server.ts` + `getAsteroidesScores` en vez de `seededScores(id.length * 17 + 3, 10)`; para el resto de `id`, sin cambios. `/juego/asteroides` ahora muestra el leaderboard real (con los datos de seed); el resto de páginas de detalle siguen igual.

4. **Modificar `app/salon-fama/page.tsx`**: cuando `tab === "asteroides"`, en un `useEffect` crear el cliente de `utils/supabase/client.ts` y llamar a `getAsteroidesScores`, guardando el resultado en estado local (`asteroidesRows`) y usándolo para el podio y la tabla en vez de `seededScores`; para el resto de pestañas, sin cambios. Mientras carga, la tabla puede mostrarse vacía/con un estado de carga breve (sin spinner dedicado, solo evitar render con `undefined`).

5. **Modificar `app/components/GamePlayer.tsx`**: al pulsar "GUARDAR PUNTUACIÓN" con `game.id === "asteroides"`, llamar a `saveAsteroidesScore` (cliente de `utils/supabase/client.ts`) con `name` y `displayScore` antes de `setSaved(true)`; el toast "PUNTUACIÓN GUARDADA" se muestra igual en ambos casos (éxito o error logueado). Para el resto de juegos, `setSaved(true)` sin llamada a Supabase, igual que hoy.

6. **Verificación final**: `npm run lint` y `npm run build` sin errores; recorrido manual — jugar una partida de Asteroides, pulsar FIN, introducir iniciales y "GUARDAR PUNTUACIÓN" (confirmar toast), luego ir a `/juego/asteroides` y a `/salon-fama` (pestaña ASTEROIDES) y verificar que la nueva puntuación aparece en el leaderboard junto a las de seed, ordenadas correctamente; confirmar que el resto de juegos/pestañas siguen mostrando `seededScores` sin cambios.

## Decisiones tomadas y descartadas

- **Supabase (Postgres) como backend de persistencia**: se descarta localStorage/JSON porque ya está configurado en el proyecto (`@supabase/supabase-js`, `@supabase/ssr`, `utils/supabase/*`) sin tablas creadas; aprovechar esa infraestructura permite un leaderboard real compartido entre jugadores, no solo local al navegador.
- **`games` como tabla de "juegos reales" (solo `"asteroides"` por ahora), no migración completa de `GAMES`**: se descarta mover los 9 juegos de `lib/data.ts` a Supabase. `games` actúa como tabla padre (FK) para `scores`, limitada a los juegos con motor real implementado. El resto del catálogo (mock) sigue viviendo en `lib/data.ts` sin tocarse.
- **Leaderboard real visible tanto en `/juego/asteroides` como en `/salon-fama`**: se descarta limitarlo solo a la página de detalle, ya que el Salón de la Fama es el lugar "canónico" de puntuaciones altas del proyecto y dejarlo con datos mock para asteroides sería inconsistente con el detalle mostrando datos reales.
- **Seed de ambas tablas (`games` y `scores`)**: se descarta lanzar con `scores` vacía para que el leaderboard de Asteroides no se vea vacío/desolador en el primer despliegue, manteniendo la estética de "tabla llena" del resto del catálogo.
- **RLS pública (select + insert) sin autenticación ni anti-cheat**: se descarta bloquear esta feature detrás de un sistema de login (fuera de alcance actual, ver spec 01). Es el equivalente persistido del comportamiento visual actual ("cualquiera" puede "guardar" su score); rate limiting/validación de rango quedarían para un spec futuro de autenticación/anti-cheat.
- **Iniciales manuales como identidad del jugador**: se mantiene el input existente del modal de fin de juego (máx. 10 caracteres, default "INVITADO") en vez de requerir una cuenta; consistente con que no hay sesión real en el proyecto.
- **Errores de inserción solo logueados, toast siempre "PUNTUACIÓN GUARDADA"**: se reutiliza el mismo patrón que el formulario de contacto (spec 04) — simplicidad de UX sobre feedback de error explícito. Riesgo aceptado: el jugador puede creer que su puntuación se guardó cuando falló silenciosamente.
- **`game.best`/`game.plays` de la entrada `"asteroides"` en `lib/data.ts` no se recalculan desde `scores`**: se descarta sincronizarlos para mantener el alcance acotado a leaderboards; esos campos siguen siendo valores estáticos mock, igual que en el resto del catálogo. Una futura mejora podría derivarlos de `scores`.
- **Sin refresco en tiempo real del leaderboard tras guardar**: se descarta suscripción realtime o revalidación automática de `/juego/asteroides`/`/salon-fama` desde el Reproductor; el jugador ve su puntuación nueva al navegar o recargar esas páginas. Mantiene el cambio acotado a una consulta simple por carga de página.
- **`ScoreRow` compartido entre datos mock y reales**: se descarta crear un tipo nuevo para las filas reales; mapear `scores` al mismo shape `ScoreRow` permite reutilizar el markup/estilos existentes de leaderboard sin condicionales adicionales de renderizado.

## Riesgos identificados

- **Migración aplicada directamente al proyecto remoto de Supabase**: no hay entorno local de Supabase configurado (`supabase/` no existe), por lo que la migración se aplica vía `apply_migration` directo al proyecto remoto. Mitigación: revisar el SQL cuidadosamente antes de aplicarlo (nombres, tipos, políticas) ya que no hay forma de probarlo en un branch local primero; si se requiere mayor seguridad, se puede usar `create_branch` de Supabase para probar la migración en una rama antes de fusionarla a `main`.
- **RLS pública de inserción en `scores`**: cualquiera con la clave pública (`NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`, visible en el bundle del cliente) puede insertar filas arbitrarias en `scores` (puntuaciones absurdas, nombres ofensivos, spam de filas). Mitigación: aceptado explícitamente para este spec (sin auth); si se vuelve un problema, un spec futuro de autenticación/anti-cheat podría restringir `insert` a usuarios autenticados y/o validar rangos de `score`.
- **React Strict Mode en `/salon-fama`**: el `useEffect` que consulta `scores` para la pestaña "asteroides" puede ejecutarse dos veces en desarrollo (doble fetch). No es un bug funcional (solo doble lectura), pero conviene verificar que no cause parpadeo visible en la tabla/podio.
- **Formato de fechas del seed**: los valores `'2026-05-12'` insertados en `created_at` (timestamptz) dependen de que `mapScoreRows` los formatee correctamente a `DD/MM/YYYY` igual que `seededScores`; un desajuste de zona horaria podría desplazar el día mostrado en ±1. Mitigación: formatear usando solo año/mes/día de la fecha (sin depender de la hora/zona horaria local del navegador).
